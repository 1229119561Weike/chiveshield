01

需求判断：这是 RAG 知识库客服，不是“训练大模型”起步

用户要把官网 FAQ、产品说明、售后政策和内部文档接入客服机器人；预算有限，核心应先验证真实问答效果。

用户原始需求

公司希望把官网 FAQ、产品说明、售后政策和内部知识文档接入一个 AI 客服机器人，能回答客户常见问题，最好能嵌入官网；预算有限，希望先验证效果，担心被高价售卖“自研大模型”或“私有化系统”。

  • 第一阶段：低成本 PoC，验证 30-50 个真实问题的命中率、无答案处理和人工转接。
  • 第二阶段：标准部署，接入官网/客服入口，增加知识库更新流程、日志、权限和文档。
  • 涉及敏感客户数据、多部门权限、CRM/工单系统联动时，再进入专业/定制级实施。
已核验开源候选11Dify、RAGFlow、AnythingLLM 等可作为采购对照。
报价梯度3 档基础 PoC、标准可用版、旗舰/专业版。
真实问题验收30-50PoC 阶段建议用真实客服问题集验证。
硬成本来源10模型 API、托管、向量库和服务器价格锚点。
02

开源候选：先用成熟组件对照供应商报价

如果供应商基于这些项目配置或二开，应披露组件、License、定制范围和交付物，不应把成熟工具包装成纯自研系统。

DifyRAG / 工作流

LLM 应用开发、RAG、工作流、可自部署；适合快速搭建知识库客服 PoC。

注意：云版和自部署成本、模型 API 成本要拆分；若供应商只是配置 Dify，应披露配置/二开范围。

RAGFlow文档 RAG

文档解析 + RAG 引擎；对 PDF/文档型知识库更友好，适合产品手册/售后政策场景。

注意：文档解析质量要用真实样本验收。

AnythingLLM轻量自托管

本地/团队知识库问答；适合中小企业内部知识库和简单客服验证。

注意：生产级客服入口和权限可能还需二开。

Open WebUI聊天 UI

多模型聊天 UI + 知识库能力；可快速验证“企业 ChatGPT 门户”体验。

注意：客服工单/渠道接入不是默认完整交付。

LibreChat多模型平台

适合企业内部多模型聊天和权限管理需求。

注意：RAG/客服渠道能力需要按实际功能评估。

Flowise可视化流程

可视化 LLM 工作流；适合让非工程人员看到 RAG/客服流程如何搭建。

注意：复杂生产系统仍需测试、日志和权限补齐。

LlamaIndexRAG 框架

数据接入、索引、RAG 框架；定制开发时可作为成熟底层框架参考。

注意：框架不是现成产品，需要工程实现。

LangChain编排生态

LLM 编排、工具调用、RAG 生态；可判断“自研 RAG 框架”是否其实是常规框架集成。

注意：需要防止过度抽象导致成本虚高。

Qdrant向量库

向量数据库；适合专业版知识库检索基础设施，可与托管/自部署成本对比。

注意：小型 PoC 未必需要独立向量数据库集群。

Chroma轻量向量库

适合 PoC 或小型知识库，降低初始部署复杂度。

注意：大规模生产能力需另评估。

Unstructured文档预处理

文档解析与预处理；可拆出 PDF/Word/网页文档清洗成本。

注意：复杂版式仍需人工抽检。

03

04

风险识别：把“听起来很 AI”的部分拆成可验收证据

风险不是为了制造甲乙方对立，而是让服务范围、成本、数据和验收边界更透明。

开源工具包装成高价定制

触发信号
不说明 Dify/RAGFlow/AnythingLLM 等组件，只说“自研 AI 客服系统”。
应要求证据
列出开源组件、定制模块、二开代码范围和 License。

模型 API 套壳冒充自研大模型

触发信号
无模型卡、无训练数据、无评测，只说“自研大模型客服”。
应要求证据
模型供应商、API 单价、上下文窗口、限流、替代模型方案。

微调过度销售

触发信号
FAQ/产品文档问答也推荐训练/微调。
应要求证据
说明 RAG、提示词、微调三种路线的成本/效果对比。

文档导入质量虚假承诺

触发信号
承诺“一键导入所有资料”但不做分块、OCR、表格/图片处理验收。
应要求证据
用真实 PDF/网页/表格样本做抽检，要求引用来源和无答案处理。

验收指标模糊

触发信号
只说“回答准确”“智能客服上线”，没有测试集。
应要求证据
约定 30-100 个真实问题、命中率、拒答率、转人工规则。

锁定与迁移风险

触发信号
不交知识库数据、向量数据、聊天记录导出方式。
应要求证据
明确源码、数据库、知识库、日志和账号归属。

数据合规风险

触发信号
客户聊天记录/订单/售后信息进入第三方模型但无说明。
应要求证据
数据脱敏、保留周期、第三方模型供应商、删除机制。

维护费陷阱

触发信号
首单便宜，后续新增文档/改提示词/换模型都另收高价。
应要求证据
约定免费维护期、更新次数、响应时间、超出单价。
05

验收清单：上线不是终点，能验才算交付

把“智能客服上线”拆成可测试问题、可追溯来源、可导出数据和可维护成本。

功能验收

✓能在测试环境访问客服入口，并完成至少 30 个真实问题问答。
✓每个回答可显示来源文档或至少能追溯知识库来源。
✓遇到知识库没有答案的问题，能明确说不知道或转人工，不胡编。
✓支持知识库新增/更新/删除，并说明多久生效。
✓支持导出知识库原文、配置和聊天记录，或明确说明不支持的原因。

技术验收

✓提供部署文档、环境变量、模型供应商和 API key 配置方式。
✓明确使用的开源项目、版本、License 和定制代码范围。
✓明确向量数据库、文件存储、日志和备份方式。
✓关键密钥不暴露在前端；后台入口有权限控制。
✓提供故障处理说明：模型超时、API 余额不足、知识库解析失败怎么办。

成本验收

✓报价拆分为需求梳理、文档清洗、配置/二开、模型 API、托管、测试培训、维护。
✓提供单次问答的 token/API 成本估算口径。
✓明确免费额度、超额调用、并发/限流和模型替换价格。
✓维护期内新增文档、改提示词、换模型、接新渠道的收费规则清楚。
06

议价话术：把该问的话写进采购沟通

这些问题不是压价，而是帮助双方把开源组件、真实定制、模型成本和验收标准讲清楚。

“这个方案里哪些部分基于 Dify、RAGFlow、AnythingLLM、LangChain/LlamaIndex 等成熟开源组件？哪些是你们真正定制开发？”
“能否把费用拆成需求梳理、文档清洗、部署配置、定制开发、模型/API 调用、服务器、维护服务？”
“为什么这个需求需要微调或训练模型？如果用 RAG + 现成模型，成本和效果差异是什么？”
“验收时我们能否用 50 个真实客服问题测试，并记录回答正确、引用正确、应转人工三类结果？”
“如果后续不续费，源码、知识库数据、向量数据、聊天记录和部署账号分别如何交接？”
“模型供应商、上下文长度、限流、单价和替代模型方案能否写进交付文档？”
“新增文档、更新知识库、修改提示词和增加渠道分别怎么收费？”
